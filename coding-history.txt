La página de inicio: ¿cuánto .md?
El inicio actual tiene secciones muy heterogéneas visualmente. En Hugo hay tres enfoques posibles:

Opción A — Front matter con bloques de datos (recomendada) El _index.md tiene front matter con los textos de cada sección, y el template los renderiza. Nada de HTML en el .md:

yaml
---
hero_titulo: "Un Hogar para Crecer."
hero_subtitulo: "Una comunidad cristiana..."
vision_texto: "Somos una congregación pentecostal..."
---
Ventaja: el editor nunca toca HTML. Desventaja: el front matter puede volverse muy largo.

Opción B — Shortcodes El .md usa shortcodes para los bloques especiales:

markdown
{{< hero titulo="Un Hogar para Crecer" >}}

{{< seccion-vision >}}

{{< eventos-recientes limite=3 >}}
Ventaja: el .md es legible y estructurado. Permite mezclar prosa real con componentes. Desventaja: hay que escribir los shortcodes.

Opción C — Template puro (como está ahora) El inicio es 100% template Hugo y no tiene .md editable salvo el título. Ventaja: control total del diseño. Desventaja: para editar copy hay que tocar HTML.

Mi recomendación: Opción B (shortcodes) para el inicio. Es el balance correcto para una iglesia donde alguien sin conocimientos técnicos tiene que poder actualizar los textos.

Taxonomías vs Tags: qué conviene para cada sección
Predicaciones — conviene una taxonomía series separada de tags de temas:

toml
# config.toml
[taxonomies]
  serie = "series"
  tag = "tags"
  tipo = "tipos"
Esto genera automáticamente páginas como /series/vida-en-el-espiritu/ listando todos los mensajes de esa serie. Los tags serían para temas transversales (espíritu-santo, familia, evangelismo).

Eventos — una taxonomía tipo (culto-especial, jovenes, evangelismo, retiro) es suficiente. Los eventos pasados y futuros se distinguen por fecha: Hugo puede filtrar con where .Date.After now.

Ministerios — no necesitan taxonomías, son pocos y fijos. Alcanza con una sección con _index.md.

2. CMS: Tina vs Decap
Decap CMS (antes Netlify CMS) es más simple de configurar, open source, no requiere cuenta externa y funciona con cualquier hosting que sirva archivos estáticos. La interfaz es básica pero funcional para alguien no técnico.

Tina CMS es más moderno, tiene edición visual en contexto (ves el cambio mientras escribís), pero requiere una cuenta en tina.io y tiene un tier gratuito con límites. Para una iglesia pequeña entraría en el free tier, pero agrega una dependencia externa.

Mi recomendación: Decap CMS. Más simple, sin dependencias de terceros, fácil de configurar con GitHub. El editor ve formularios claros para cada campo (título, predicador, fecha, link de YouTube, etc.) sin tocar markdown.

3. YouTube como fuente de audio/video
Es la decisión correcta. El .md guarda solo el ID o la URL del video:

yaml
youtube: "dQw4w9WgXcQ"
En el template se renderiza como un embed o como un link con thumbnail. Hay dos opciones de presentación a decidir:

A) Embed directo — el video de YouTube aparece dentro de la página de la predicación. Ventaja: el visitante no sale del sitio. Desventaja: carga pesada, privacidad (cookies de Google).

B) Link con thumbnail — muestra la miniatura del video con un botón "Ver en YouTube" que abre en nueva pestaña. Más liviano y más honesto.

Para una iglesia recomendaría B para el listado y A disponible en la página de detalle de cada predicación, donde el visitante ya eligió ver ese mensaje en particular.

Lo que pasó después, si hay algo que contar.
Para el home: mostrar solo los próximos 3 eventos futuros. Si no hay ninguno, mostrar un mensaje tipo "Seguinos en Instagram para novedades" o directamente ocultar la sección. Mostrar eventos pasados en el home no tiene mucho valor para un visitante nuevo.

Para la página /eventos/: dos secciones separadas — próximos arriba, historial abajo colapsado o con paginación.

Sobre la crónica post-evento: que viva en el mismo .md es la solución más simple. El editor entra al archivo, cambia estado: "realizado" y agrega una sección ## Crónica con el texto. No requiere crear un nuevo archivo ni vincular nada. Si no hay novedad, el campo queda vacío y el template no muestra esa sección.

Instagram Stories/Highlights: ¿conviene?
Respuesta corta: no las embebas directamente, pero sí podés vincularlas.

Las razones:

Las Stories desaparecen a las 24 horas. Los Highlights sí son permanentes, pero la API de Instagram para embeds es restrictiva y cambia seguido (Meta ha roto embeds de terceros varias veces).
Cualquier solución de embed que funcione hoy puede dejar de funcionar en 6 meses sin aviso.
El peso no es el problema principal: es la fragilidad de la dependencia.
Lo que sí tiene sentido: un botón o sección en la página que diga "Seguinos en Instagram" con link directo al perfil o a un Highlight específico. Simple, sin roturas, sin dependencias. El visitante que quiere ver las fotos del evento va a Instagram, que es donde ya las tiene.

Si en el futuro querés algo más integrado, herramientas como Behold.so o Elfsight hacen widgets de feed de Instagram que son más estables, pero son servicios pagos.
